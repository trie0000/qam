# SPO 共有化（複数人同時利用）設計方針

QAM の管理データを SharePoint Online（以下 SPO）のリスト／ドキュメントライブラリへ移し、
**複数人が同時に使える**ようにするための設計。UI は SharePoint ページへ overlay 注入する
bookmarklet 方式に変え、起動は CDP（Chrome DevTools Protocol）方式に寄せる。
同構成の別ツールで実績のある方式を踏襲する。

> 本書は方針であって手順書ではない。実装時の詳細は各フェーズで `DESIGN.md` へ落とす。

---

## 0. 目的と前提

**目的**

- 取り込んだ資産・履歴・メモを**全員が同じものを見る**（今は各自のローカル/共有フォルダ依存）
- 端末を入れ替えてもデータが消えない
- 「誰がいつ何をしたか」を SPO の作成者・更新日時で自然に残す

**前提**

- SPO テナントと、**既存サイトへの相乗り**（リスト作成権限あり）
- **取り込みは全員が行う**（担当者限定ではない）
- Qualys 認証情報は**完全に個人設定**。共有ストアには置かない
- 秘密情報をリポジトリ・SPO・ログのいずれにも書かない（`CLAUDE.md` の方針の延長）

---

## 1. 全体構成

relay は無くならない。**Qualys API は CORS 非対応・Basic 認証・社内プロキシ経由**のため、
SharePoint ページ内の JS から直接は叩けない。relay が担う 2 役のうち片方だけを SPO へ移す。

| 役割 | 現在 | 移行後 |
|---|---|---|
| ファイル保管 | relay（ローカル/UNC のデータディレクトリ） | **SPO（リスト＋ライブラリ）** |
| Qualys 中継 | relay | **relay（据え置き）** — データディレクトリを持たない中継に痩せる |

```
起動バッチ → ランチャ(.ps1)
  1) relay の /qam/health を待つ
  2) 専用プロファイルの Edge を起動（--remote-debugging-port / --remote-allow-origins=*）
  3) 認証完了を待つ（_api/web への fetch が通るまでポーリング）
  4) CDP の Runtime.evaluate でローダを注入 → SP ページに overlay としてマウント
  5) 失敗時は従来フロー（ローカル relay の画面）へフォールバック
```

- ヘッドレス不可・専用プロファイルは `%LOCALAPPDATA%` 配下・既存プロファイルは移動しない
  （移動すると全員が再サインインになる。実績のある踏み方に従う）
- アプリは**認証情報に一切触らない**。CDP は「認証済みページに JS を注入する」だけなので
  MFA / SSO / 条件付きアクセスはそのまま機能する
- SP REST は同一オリジン Cookie（`credentials: 'include'`）で叩く

## 2. データ配置

### 2.1 ドキュメントライブラリ `QamData`（大きい・不変・まとめて読む）

| 現在のパス | 移行後 | 理由 |
|---|---|---|
| `snapshots/<entity>/<stamp>.json` | 同パス | **取込ごとに新規ファイル＝不変**。同時実行でも競合しない。stamp が不変なのでクライアント側に永続キャッシュできる |
| `history/<entity>.jsonl` | `history/<entity>/<YYYY-MM>.jsonl` | SPO のファイル更新は全文上書き。月分割で書き戻し量を抑える（1年12ファイル＝全期間読んでも 12 リクエスト） |
| `inspection/<date>.json` | 同パス | 四半期検査の生 XML キャッシュ |
| `raw/`, `backups/` | 同パス | SPO のバージョン履歴・ごみ箱がそのままバックアップになる |

既存の `FileBackend`（read / write / list / remove）を**ライブラリ用アダプタとして残す**。
パス互換なので `store.ts` のスナップショット・履歴まわりはほぼ無改修で載る。

### 2.2 リスト（複数人が同時に足す・直す）

すべて `Qam` プレフィックス（既存サイトへの相乗りなので名前衝突を避ける）。

| リスト | 用途 | 要点 |
|---|---|---|
| `QamComments` | 資産ごとの作業メモ | `TargetId` をインデックス列に |
| `QamAnnotations` | 資産ごとの注釈 | `Entity+TargetId+Field` の DedupKey に**一意制約** |
| `QamOps` | 操作履歴 | 作成者・作成日時は SPO が自動で持つ |
| `QamInspections` | 簡易検査の管理表 | 申請情報 ＋ provision スナップショット（Note 列） |
| `QamLicenses` | ライセンス数推移のサンプル | |
| `QamSettings` | **共有設定** ＋ **排他クレーム行** | `SettingKey` に**一意制約**（取込ロックの土台） |

`QamAnnotations` を列挙しているのは、現在 `annotations/<entity>.json` が
**エンティティごと 1 ファイル**で、複数人が同時に注釈を付けるとロストアップデートになるため。

### 2.3 共有設定と個人設定の切り分け

| 区分 | 項目 | 置き場所 |
|---|---|---|
| 共有 | 保存期間・年度開始月・接続点ID パターン・オプションプロファイル・スキャナー・タイムゾーン・地域区分・ライセンス上限・**プロキシ URL**・接続先 POD | `QamSettings` |
| 個人 | 記入者名、表示設定（列幅・フィルタ・テーマ）、Qualys ユーザー ID | **localStorage（平文）** |
| 個人・秘密 | **Qualys パスワード** | **localStorage（DPAPI 暗号文）** — §4 |
| ブートストラップ | relay のポート、データディレクトリ（移行期のみ） | env（起動時に必要で SPO を読む前に要るものだけ） |

**設定項目の追加は SPO 側の列追加**（`ensureLists`）で全員に同時に配られる。
env ファイルを配布する工程が消える。

## 3. 同時利用の競合制御

**取り込みは全員が行う**前提なので、4 層で守る。

| 層 | 対象 | 方法 |
|---|---|---|
| 1. 不変 | スナップショット | stamp 単位の新規ファイル → そもそも競合しない |
| 2. 追記 | コメント・注釈・操作履歴・管理表 | **リストへの POST**。SPO が採番するのでロストアップデートが起きない |
| 3. 更新 | 既存行の編集 | **MERGE + `If-Match: <etag>`** → 412 なら再読込して再適用 |
| 4. 排他 | 取り込み | `QamSettings` の一意キー行を claim（SPO の `EnforceUniqueValues` がアトミックな mutex） |

### 3.1 「追記だけだから安全」は成り立たない

SPO のファイル更新には**追記 API が無く、常に全文上書き**である。アプリ側の「追記」は
実際には `読む → 足す → 全文書き戻す` の 3 手なので、更新も削除も無くても

```
A: 読む(S0) ── 足す ──────── 書き戻す(S0+A)
B:   読む(S0) ── 足す ── 書き戻す(S0+B)     ← A の行が消える
```

が起きる。危険は「何を書くか」ではなく「読んでから書くまでに他人が書けるか」で決まる。

ただし**追記専用であることには意味がある**。競合の *検出* さえできれば、*解消* は自明になる
（読み直して自分の分を足すだけ）。3-way マージのような機構は要らない。

#### ★実機検証の結果（2026-07-19・当初の想定は誤りだった）

当初は「履歴ファイルも `If-Match` で検出 → リトライで収束する」と設計したが、**実機で否定された**。

| 対象 | `If-Match` | 実測 |
|---|---|---|
| **リスト項目**（MERGE） | **効く** | 不正な ETag → **412**。正しい ETag → 204 |
| **ファイル**（`$value` の PUT） | **効かない** | 不正な ETag でも **204** で通り、**内容が上書きされる** |

つまり**ファイルには条件付き書き込みが無く、競合の検出自体ができない**。
`Files/add(overwrite=true)` も同様に無条件。したがって、

- **記録系（メモ・注釈・操作履歴・管理表・ライセンス推移）はリストに置く**。ここは If-Match が効くので
  Phase 1 の設計どおりで問題ない
- **ファイルの追記は「取込ロックで直列化する」ことで守る**（§3.2）。SP 保管でファイルを追記するのは
  取込経路だけなので、ロック下に収まっている
- コードは If-Match を送り続けるが、**それに依存しない**（将来 SP の挙動が変わったときの保険）

残る弱点: ロックは TTL で失効するため、**極端に長い取込が失効後の別取込と重なると履歴の追記が
後勝ちになりうる**。これを完全に無くすには履歴を「取込ごとの不変ファイル」にする（読む→足す→書く
自体を無くす）のが正しい。未対応。

### 3.2 取込ロックが本当に防ぐもの

データ破損は §3.1 の `If-Match` で防げる。ロックが防ぐのは**重複取込**である。
2 人が同時に取り込むと、両者とも「前回スナップショット S0 との差分」を計算するため、

- ほぼ同一のイベントが 2 重に履歴へ入る
- 数秒違いのスナップショットが 2 本できる
- Qualys から数千件を 2 回引く（所要時間も API 消費も倍）

対策は 2 段構え。

- **取込ロック**: claim 行に所有者と開始時刻を持たせ、他の人には「◯◯さんが取込中です（12:34 開始）」と表示。
  **TTL（15 分程度）で自動失効**させる（ブラウザを閉じてロックが残る事故を防ぐ）＋ 手動解除も用意
- **直近取込ガード**: 「最新の取込は 12:34（◯◯さん）です。再取得しますか？」と確認してから走らせる。
  全員が取り込む運用では体感的にこちらの方が効く

他者の取込が完了したら、**再取得ではなく画面のリロードで済む**導線にする
（誰かが取れば全員に反映されるのが共有化の主目的のため）。

## 4. 認証情報・個人設定の扱い

### 4.1 方針

Qualys パスワードは **localStorage に DPAPI 暗号文で置く**。平文はブラウザにもファイルにも残さない。

```
① 設定画面でパスワードを入力
② relay の /qam/secret/protect へ POST
③ relay が DPAPI（現在の Windows ユーザースコープ）で暗号化し、暗号文だけ返す ← 平文はここで捨てる
④ ブラウザは暗号文を localStorage に保存
⑤ Qualys 呼び出し時は暗号文を relay へ渡し、relay 内部だけで復号して使う
```

**`unprotect` エンドポイントは作らない。** 作ると平文を取り出せてしまい、この方式の意味が消える。
復号は Qualys へ送出する直前に relay 内部でのみ行う。

### 4.2 なぜこの形か（検討の記録）

localStorage は**オリジン単位**であり、SharePoint のチームサイトは
**サイトが分かれていてもオリジンが 1 つ**である。

```
https://YOUR-TENANT.sharepoint.com/sites/A ┐
https://YOUR-TENANT.sharepoint.com/sites/B ├─ すべて同一オリジン = 同一 localStorage
https://YOUR-TENANT.sharepoint.com/sites/C ┘
```

したがって平文で置くと、**テナント内のどの SharePoint ページで動くスクリプトからでも読める**。
現在の保存先は relay 自身が配信する画面のオリジンだけなので、ここは明確な後退になる。

一方、**env ファイルへ移しても平文である事実は変わらない**。置き場所の違いで決まるのは
「誰が読めるか」であって、「読まれたときに使えるか」ではない。むしろ env は

- 配置場所が Desktop / Documents 配下だとクラウド同期で machine の外へ出る
- relay 一式を共有フォルダから動かすと共有権限を持つ全員に見える

という経路があり、安全側とは言えない。よって**軸を分け**、置き場所は要望どおり
localStorage に集約したうえで、**平文をやめる**ことで解決する。

| | 同一ユーザーのプロセス | 同期・バックアップで外へ | SP ページのスクリプト | 盗まれた後 |
|---|---|---|---|---|
| env 平文 | 読める | 出る場合がある | 読めない | そのまま使える |
| localStorage 平文 | 読める | 出る場合がある | **読める** | そのまま使える |
| **localStorage + DPAPI** | 読めるが無意味 | 出ても無意味 | 取れるが無意味 | **他 PC・他ユーザーでは使えない** |

残る穴は「同一 PC の別ページが暗号文を relay へ渡して Qualys を叩かせる」ことなので、
**relay の CORS を対象テナントの URL に限定する**（現在の `*` は絞る）。

### 4.3 付随ルール

- 暗号文は Windows ユーザー＋マシンに紐づく。PC を替えたら再入力（仕様として案内する）
- **監査ログ・relay.log に秘密を書かない**。`pass` / `secret` はログ対象から明示的に除外する
- 「保存しない」も選べるようにする（relay のメモリのみ／起動ごとに 1 回入力）

## 5. コード構造

```
src/api/
  repo.ts        Repository インタフェース（comments / annotations / ops / inspections / licenses / settings）
  sp.ts          SP REST 実装（digest / ETag / $batch）
  sp/schema.ts   FieldSpec を集約。ensureLists() で作成・列追加（SCHEMA_VERSION でバンプ）
  mock.ts        SPO 無しで UI を検証
  local.ts       現行 relay ファイル実装（フォールバックとして維持）
```

- 既存 `FileBackend` はライブラリ用アダプタとして残す（§2.1）
- 未 migrate 環境へ新列を無条件に送らない。**値があるときだけ body に載せ、`SCHEMA_VERSION` を上げる**
- バンドルは SPO のライブラリに置き、ローダがサイレント更新する

## 6. 段階計画

各フェーズで前の状態に戻せることを条件にする。

| Phase | 内容 | 状態 |
|---|---|---|
| 0 | 保管先の差し替え口（`setBackend`）＋ ライブラリ実装（`api/sp-file.ts`）。設定 `storageMode` で local / sp を切替。接続できなければ起動時にローカルへ自動フォールバック | **実装済み** |
| 1 | リスト実装（`sp/schema.ts` ＋ `ensureLists`）。コメント / 注釈 / 操作履歴 / 管理表 / ライセンス推移をリストへ移す（共有設定は Phase 2 で取込ロックと同時に） | **実装済み** |
| 2 | 取込ロック（一意キー行 + TTL）＋ 直近取込ガード | **実装済み**（ローカル→SPO 移送は未着手） |
| 3 | 配布（バンドルを SPO へ、ローダ、CDP ランチャ）＋ relay の痩せ替え ＋ DPAPI 秘密管理 | 未着手 |

### Phase 0 で入った範囲

- `src/api/sp-file.ts` — SharePoint ライブラリを `FileBackend` として実装。`store.ts` はパスでしか
  物を見ないので、**この差し替えだけで全データ（スナップショット・履歴・コメント・注釈・操作履歴・
  管理表・ライセンス推移）が SPO 上の共有になる**（`store.ts` は無改修）
- **追記は `If-Match` + 無条件リトライ**（§3.1）。新規作成は `overwrite=false` で、
  同時作成を上書きしない
- 追記でない書き込み（スナップショット等の不変ファイル・注釈の全体書き換え）は
  `overwrite=true` の add。**注釈のロストアップデートは Phase 1 のリスト化で解消する**
- 起動時に一度だけ疎通を確認し、失敗したらローカル保管へ戻して警告する
  （切替ミスで「データが空」に見える事故を防ぐ）

### Phase 1 で入った範囲

- `api/repo.ts` — 記録系（メモ / 注釈 / 操作履歴 / 管理表 / ライセンス推移）の `RecordRepo`。
  ファイル実装（store.ts へ委譲）と SharePoint リスト実装を差し替える
- `api/sp/list.ts` — リスト/列の自動作成（`ensureList`）、`__next` 追従の全件取得、
  追記（POST）、更新（MERGE + `If-Match`）、削除
- `api/sp/schema.ts` — 5 リストの列定義と QAM 型との相互変換。`SCHEMA_VERSION` は列追加で上げる
- `api/sp-repo.ts` — 注釈は `DedupKey`（`entity|id|field`）の**一意制約**で「資産×項目 1 行」を保証。
  同時作成は一意制約で弾かれるので更新に回し、412 は読み直して再適用する
- **ファイル実装の全文書き戻し（注釈・メモ編集）の競合はこれで解消**。追記もリストへの POST になり、
  SPO が採番するのでロストアップデートが起きない
- 共有設定（`QamSettings`）は Phase 2 で取込ロックと同時に入れる（`getConfig`/`setConfig` の
  置き換えを伴うため、ロックの排他クレーム行と一緒に設計した方が無駄がない）

### Phase 2 で入った範囲

- `QamSettings` リストを追加。`SettingKey` の**一意制約**が、同時に取りに来ても
  **行を作れるのは 1 人だけ**という原子的な mutex になる
- **取込ロック**（`acquireIngestLock` / `releaseIngestLock`）:
  - 取れたら `null`、他の人が取込中ならその保持者（誰が・いつから）を返す
  - **TTL 15 分**で自動失効。ブラウザを閉じたまま放置されても詰まらない
  - 期限切れの行は `If-Match` 付きの更新で引き継ぐ（同時なら片方だけ成功）
  - 解放は**自分の行だけ**。引き継がれた後に遅れて解放しに来ても他人のロックを消さない
  - 手動取込・無人取込（タスクスケジューラ）の両方に掛ける。無人側は非対話なので
    取れなければ静かに諦め、操作履歴に理由を残す
- **直近取込ガード**: 30 分以内に誰かが取り込んでいたら「最新の取込は 12:34（◯◯さん）です。
  取り込み直しますか？」と確認する。判定できないこと（履歴が読めない等）を理由に取込は止めない
- ファイル保管は 1 人用なのでロックは素通し（共有フォルダでも原子的な排他手段が無いため、
  中途半端に効くふりをしない）

**共有設定の SPO 移行（`getConfig`/`setConfig` の置き換え）とローカル→SPO 移送は未着手。**

## 7. 着手前に潰す検証

> **§7 の 2・3・4 のうち、リスト/ライブラリの REST 契約は実機で確認済み（下表）。**
>
> | 確認項目 | 結果 |
> |---|---|
> | 要求ダイジェスト（`contextinfo`） | 取得可・TTL 1800 秒 |
> | リスト作成 / 列作成（`FieldTypeKind` 2/3/9） | 201 |
> | 列の内部名 | **表示名と一致**（空白なし ASCII なら `_x0020_` 化されない） |
> | `EnforceUniqueValues` の付与 | 204。**重複 add は 500 で弾かれる**（＝取込ロックの mutex が成立） |
> | リスト項目の ETag / `If-Match` | `"1"` 形式。不正なら **412**、正しければ 204 |
> | ファイルの `$value` GET | **ETag ヘッダは返らない**。プロパティ `?$select=ETag` から取得（`"{GUID},<版>"`） |
> | ファイルの `$value` PUT の `If-Match` | **無視される**（§3.1 参照） |
> | 未存在フォルダの `?$select=Exists` | **200 + `Exists:false`**（404 ではない） |
> | `Files/add(overwrite=false)` の既存衝突 | 400 で弾かれる |

1. **SP ページのオリジンから `http://127.0.0.1:<port>` への fetch が通るか。**
   CORS に加え Private Network Access のプリフライトが要るかを実機で確認する。
   あわせて `Access-Control-Allow-Origin` をテナント URL へ限定する。
2. **数 MB の host スナップショットの読み書き実効速度。** stamp が不変なので
   IndexedDB へ永続キャッシュできるが、初回の体感を測る。
3. **リストの 5000 件ビューしきい値。** `QamOps` / `QamComments` が超えたときの
   インデックス列と `$filter` 設計を先に決める。
4. **DPAPI の実装可否。** `ConvertFrom-SecureString` / `ConvertTo-SecureString`（`-Key` 無し）は
   Windows 上の PowerShell 5.1 / 7 の双方で DPAPI を使う。**Windows 実機での確認が必要**
   （非 Windows では利用できない）。

---

## 8. 得るもの・失うもの

**得る**: 端末を選ばない／PC 入替でデータが消えない／作成者・更新日時が SPO 側に自動で残る／
バックアップが SPO 標準機能に寄る／設定項目の追加がファイル配布を伴わない。

**失う**: 起動と一覧表示に SP REST のラウンドトリップが乗る／SPO の制約
（ビューしきい値・列型・Note 列の文字数上限）に付き合う／完全オフラインでは使えない。
